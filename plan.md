# AI 자율 암호화폐 매매 실험 — 최종 계획

> 실험 개요: Claude에게 실제 자금 10만원을 맡기고 업비트 KRW 마켓에서 자율 매매시킨다.
> 매주 평가하여 **증식 성공 시** 자금 증액 + 전략 메모리 보존 + 권한 확대, **실패 시** 전략 메모리 초기화("kill") 후 기본값으로 재시작.
> Claude는 프롬프트를 통해 자신의 생존 조건을 인지한 상태로 매매한다 (실험 컨셉 유지).

---

## 1. 전체 아키텍처

```
              ┌────────────────────────────────────────────────────────────┐
              │                    스케줄러 (APScheduler)                    │
              └──┬───────────┬──────────────┬──────────────┬───────────────┘
                 │ 동적 주기   │ 1분마다       │ 매시 정각     │ 주 1회 (월 09:00)
                 ▼           ▼              ▼              ▼
        [트레이딩 사이클]  [청산 감시]     [자산 스냅샷]   [주간 평가 Judge]
                 │        (LLM 없음)         │              │
 1. 시세/지표 수집 │      손절 -7%            │   수수료 차감 후 순평가액 비교
 2. 상태 로드     │      익절 +15% (절반)     │   ├─ 성공: 자금 증액 + 메모리 보존 + 권한 확대
 3. 예산 확인 ────┤      트레일링 -5%        │   └─ 실패: 전략 메모리 초기화 (거래 로그는 보존)
 4. Claude 판단   │      서킷브레이커 -15%    │
 5. 가드레일 검증 →│           │              │
    주문 실행      ▼           ▼              ▼
 6. 로그 + 다음  ┌──────────────────────────────┐
    사이클 예약 ▶│      SQLite + 파일 저장소       │ ◀── [웹 대시보드 (FastAPI)]
                └──────────────────────────────┘      읽기 전용, PC/모바일
```

> **핵심**: 청산 감시 잡은 Claude를 호출하지 않는 순수 규칙 엔진이다. 따라서 **API 비용이 0원**이며, 예산 소진으로 판단이 멈춰도 손절·익절·서킷브레이커는 계속 포지션을 지킨다.

### 핵심 설계 원칙

| 원칙 | 내용 |
|---|---|
| 사이클 ≠ 매매 | LLM의 기본 응답은 "보유". 매매는 필요할 때만 발생 |
| 동적 스케줄링 | LLM이 매 사이클 종료 시 다음 체크 시점을 직접 지정 (최소 1시간 ~ 최대 24시간) |
| LLM은 방향, 엔진은 실행 | LLM은 목표 비중만 결정. 주문 수량 계산·타이밍·검증은 규칙 기반 엔진 담당 |
| **청산은 LLM과 분리** | 손절·익절·서킷브레이커는 1분 주기 규칙 엔진이 담당. LLM 사이클 간격(최대 24h)에 손절이 묶이지 않도록 |
| kill 프레이밍 유지 | Claude는 생존 조건을 알고 매매하되, 도박성 행동은 가드레일이 기계적으로 차단 |
| 대시보드는 읽기 전용 | 웹/모바일에서 조회만 가능. 주문 실행 경로는 봇 내부에만 존재 |

## 2. 가드레일 (executor가 기계적으로 강제)

| # | 가드레일 | 기본값 | 목적 |
|---|---|---|---|
| 1 | 평가 전 쿨다운 | 주간 평가 24시간 전부터 신규 매수/비중 확대 차단 (매도·보유만 허용) | 막판 몰빵 도박 방지 |
| 2 | 단일 코인 집중도 상한 | 최대 50% | 상시 "한 방 노리기" 구조적 제한 |
| 3 | 최소 변경 임계치 | 목표-현재 비중 차이 5%p 미만이면 주문 생략 | 잔손질 수수료 누수 차단 |
| 4 | 일일 매매 횟수 상한 | 6건/일 (예산제, LLM에게 잔여량 공지) | 과매매 억제 |
| 5 | 일일 손실 서킷브레이커 | 당일 -15% 도달 시 당일 매매 전면 중단 + 알림 | 급락장 손실 확산 방지 |
| 6 | 1회 최대 주문 금액 | 운용자금의 30% | 단일 주문 사고 제한 |
| 7 | 거래 대상 유니버스 제한 | KRW 마켓 24h 거래대금 상위 20개 코인만 | 저유동성 잡코인 슬리피지/조작 위험 회피 |

**위반 시 동작은 세 가지로 나뉜다** (구현하며 조정 — 초안의 일괄 "거부"는 부작용이 있었다):

| 동작 | 해당 가드레일 | 이유 |
|---|---|---|
| **거부(reject)** | #1 쿨다운 중 매수, #4 일일 한도, #5 서킷브레이커, #7 유니버스 밖 | 실행하면 안 되는 주문. 통째로 버린다 |
| **조정(clamp)** | #2 비중 상한, #6 1회 주문금액, 비중 합 >100%, 손절선 범위 | 의도는 살리되 한도까지 깎는다 |
| **생략(skip)** | #3 최소 변경 임계치, 최소주문금액 미만 | 주문할 이유가 없다 |

- 조정을 도입한 이유: "BTC 70%"를 통째로 거부하면 BTC 비중이 **0%**가 되어 오히려 Claude의 의도에서 더 멀어진다. 50%로 깎는 쪽이 안전하면서 의도에 가깝다.
- 거부·조정 내역은 **모두 다음 사이클 컨텍스트로 피드백**되어 Claude가 규칙을 학습한다.
- 모든 한도값은 `config.py`에서 관리. 주간 평가 성공 시 "권한 확대"는 이 한도값 일부를 완화하는 방식 (레벨당 일일 매매 +2건, 유니버스 +5개, 운용 한도 +5만원, 최대 레벨 3).

## 3. 자동 청산 규칙 (`agent/watchdog.py`)

**1분 주기 감시 잡. Claude 호출 없음 = API 비용 0원.** 현재가만 조회해 규칙 위반 시 즉시 시장가 청산한다.

| # | 규칙 | 기본값 | 동작 |
|---|---|---|---|
| 1 | 고정 손절 (stop-loss) | 진입 평단 대비 **-7%** | 해당 코인 **전량** 시장가 매도 |
| 2 | 익절 (take-profit) | 진입 평단 대비 **+15%** | **절반만** 매도, 잔량은 트레일링으로 전환 |
| 3 | 트레일링 스탑 | 익절 발동 후 고점 대비 **-5%** | 잔량 전량 매도 |
| 4 | 일일 서킷브레이커 | 당일 총평가액 **-15%** | 전 포지션 청산 + 당일 신규 매매 전면 중단 |

**설계 근거**
- **익절을 전량이 아닌 절반으로 하는 이유**: 크립토는 추세가 길게 이어지는 경우가 있어 +15%에서 전량 청산하면 상방을 자른다. 절반 익절 + 트레일링으로 수익을 연장하되 원금은 확보.
- **평단 기준**: 분할 매수 시 가중평균 매입단가(평단)를 기준으로 계산. `positions` 테이블에 코인별 평단·최고가를 유지.
- **청산 사유는 다음 사이클 컨텍스트에 전달**: "BTC가 -7% 손절로 강제 청산됨"을 Claude가 알아야 다음 판단이 성립한다.
- **Claude의 손절선 조정 권한**: 매수 시 코인별 손절선을 직접 지정할 수 있으나 config 한계(**-10% 이내**) 안에서만. 미지정 시 기본값 -7% 적용.
- **최소주문금액 예외**: 청산 후 잔량이 5,000원 미만이면 분할 매도 대신 전량 매도로 처리 (업비트 최소주문금액 제약).

## 4. API 예산 관리 (`agent/budget.py`)

**월 상한 $10. 단, 상한 도달 시 즉사가 아니라 3단계로 감속한다** — 월 중순에 죽으면 남은 기간 실험이 통째로 멈추기 때문.

| 소진율 | 상태 | 동작 |
|---|---|---|
| ~80% | `normal` | 정상 운영 (effort=high, 최소 사이클 간격 1시간) |
| 80~100% | `throttled` | **자동 감속**: effort를 `medium`으로 하향, 최소 사이클 간격 4시간으로 강제 |
| 100% ↑ | `suspended` | **Claude 호출 정지**. 청산 감시 잡은 계속 작동(포지션은 규칙이 보호). 매월 1일 00:00 KST 자동 리셋·재개 |

- **실측 기반 집계**: 매 호출의 `response.usage`(입력/출력/캐시 생성/캐시 읽기 토큰)를 모델 단가로 환산해 `api_usage` 테이블에 누적. 추정치가 아닌 실제 과금 기준.
- **사전 차단**: 사이클 시작 전 예산을 확인하고, 남은 예산이 1회 호출 예상 비용보다 적으면 호출 자체를 건너뛴다 (초과 과금 방지).
- **정지 중에도 살아있는 것**: 청산 감시(§3 전체), 자산 스냅샷, 주간 평가(judge — 규칙 기반이라 LLM 미사용), 대시보드.
- **정지 중 멈추는 것**: 신규 매수·비중 조정 등 Claude의 모든 판단. 즉 "보유 + 규칙 청산" 상태로 월말까지 대기.
- 상태 전환 시 로그 기록 + 대시보드 배지 표시 (§5.2 ①).

## 5. 주간 평가 (Judge) 규칙

- **판정 기준**: 주간 시작 스냅샷 대비 종료 시점 총 평가액 (수수료·슬리피지 모두 반영된 실제 잔고 기준).
- **성공** (순수익 > 0):
  - 운용자금 증액 (증액 규칙: 사용자 결정 필요 — §10 참고)
  - 전략 메모리 보존
  - 권한 1단계 확대 (연속 성공 시 누적)
- **실패** (순수익 ≤ 0):
  - `strategy_memory` 초기화 ("kill") — 거래 로그·스냅샷은 보존 (역사 확인용)
  - 운용자금·권한 기본값 복귀
- kill/성공 이벤트는 별도 테이블에 기록되어 대시보드에 "세대(Generation) 히스토리"로 표시.

## 6. 웹 대시보드

### 6.1 기술 선택

- **백엔드**: FastAPI — 봇과 같은 프로세스/저장소(SQLite)를 공유하는 읽기 전용 API.
- **프론트엔드**: 서버 렌더링 HTML + vanilla JS + Chart.js. 별도 빌드 도구 없음.
- **모바일**: 반응형 레이아웃 + PWA manifest → 폰 홈 화면에 앱처럼 추가 가능. 네이티브 앱 불필요.
- **접근 보안**: 1인 사용이므로 두 가지 옵션 중 택 1
  - **(권장) Tailscale VPN**: 대시보드를 공인 인터넷에 아예 노출하지 않음. VPS와 폰에 Tailscale 설치만 하면 끝. 인증 코드 작성 불필요.
  - 공개 노출 시: 토큰 기반 로그인 + Caddy 리버스 프록시로 HTTPS 자동화.

### 6.2 화면 구성

**① 홈 (요약)**
- 수익률 카드 4개: **오늘 / 주간 / 월간 / 전체** (원금 대비 %, 금액)
- 총 평가액 + 자산 곡선 차트 (시간별 스냅샷 기반, 기간 전환: 1일/1주/1개월/전체)
- 현재 포트폴리오 도넛 차트 (코인별 비중, 코인별 평가손익, **코인별 손절선까지 남은 거리 %**)
- 상태 배지: DRY-RUN 여부 / 서킷브레이커 발동 여부 / 평가 전 쿨다운 여부 / 다음 사이클 예정 시각 / 오늘 매매 예산 잔여
- **API 예산 위젯**: 이번 달 소진액 / $10 (진행 바), 상태(`normal`·`throttled`·`suspended`), 정지 시 재개 예정일

**② 거래 내역**
- 시간순 리스트: 일시, 매수/매도, 코인, 수량, 체결가, 금액, 수수료
- **매도 사유 태그**: `LLM 판단` / `손절 -7%` / `익절 +15%` / `트레일링` / `서킷브레이커` — 규칙 청산과 판단 매도를 한눈에 구분
- 각 거래에 **LLM의 판단 사유 전문** 첨부 (탭하면 펼침) — "왜 샀는지/팔았는지"를 그대로 열람
- 가드레일에 의해 거부된 주문도 별도 표시 (거부 사유 포함)
- 필터: 기간 / 코인 / 매수·매도 / 거부 포함 여부 / 청산 사유

**③ AI 두뇌 (전략 메모리)**
- 현재 `strategy_memory` 전문 뷰어 — Claude가 스스로 남긴 관찰/계획 노트
- 사이클별 판단 로그 타임라인 (매매 없이 "보유"로 끝난 사이클의 사유도 포함)

**④ 세대 히스토리**
- 주간 평가 결과 테이블: 주차, 시작/종료 평가액, 수익률, 성공/실패, kill 여부, 당시 권한 레벨
- 세대별 생존 기간 시각화 (몇 주 살아남았는지)

**⑤ 청산 감시 상태**
- 코인별 진입 평단 / 현재가 / 손절선 / 최고가 / 트레일링 발동 여부
- 감시 잡 마지막 실행 시각 (죽어 있는지 즉시 확인 가능)

### 6.3 지원 데이터 (스케줄러에 잡 추가)

- **자산 스냅샷 잡 (매시 정각)**: 총 평가액을 SQLite에 기록. 오늘/주간/월간 수익률과 자산 곡선의 원천 데이터.
  - 오늘 수익률 = 현재 평가액 ÷ 당일 00:00(KST) 스냅샷 − 1
  - 주간/월간도 동일 방식. 스냅샷 누락 시 가장 가까운 이전 스냅샷으로 대체.

## 7. 기술 스택 (확정)

| 영역 | 선택 | 비고 |
|---|---|---|
| 언어 | **Python 3.10** | 이 PC에 설치된 버전. 필요한 기능이 전부 3.10에 있어 3.12 요구를 철회 |
| 거래소 | **`requests` + `PyJWT` 직접 REST** | pyupbit 대신. 공개 시세는 무인증, 주문만 JWT 서명. 래퍼 의존 없이 재시도·에러 처리를 직접 제어 |
| LLM | `anthropic` SDK, **`claude-sonnet-5`** | tool use로 JSON 스키마 강제. adaptive thinking + effort(`high`), 프롬프트 캐싱 적용 |
| 스케줄러 | APScheduler | 동적 주기 + 1분 감시 잡 필요하므로 OS 스케줄러 부적합 |
| 저장소 | SQLite (거래/스냅샷/평가/포지션/API사용량) + `strategy_memory.md` (사람이 읽는 전략 노트) | |
| 웹 | FastAPI + Jinja2 + Chart.js + PWA | 빌드 도구 없음, 읽기 전용 |
| 설정 | pydantic-settings + `.env` | 한도값 오타 방지 |
| 로깅 | loguru | 파일 로테이션 |

**LLM 비용 추산**: 하루 ~5사이클 기준 월 $4–5 예상 (캐싱 적용 시 더 낮음). 월 상한 $10은 정상 운영 대비 약 2배 여유 → `throttled` 진입은 드물 것으로 예상되나, 안전장치로 유지.

## 11. 검증 계획

1. **단위 테스트**: judge 성공/실패 분기, 가드레일 7종 각각, **청산 규칙 4종**(손절·익절 절반·트레일링·서킷브레이커), **예산 상태 전환**(80%/100% 경계), 수익률 계산(스냅샷 경계 케이스), DRY_RUN 모의 체결 로직.
2. **dry-run 1–2주**: 실제 시세로 모의 운용. 대시보드에서 판단 품질 직접 관찰.
3. **서킷브레이커 강제 트리거**: 모의 잔고를 인위 조작해 -15% 상황을 만들어 자동 중단 확인.
4. **청산 규칙 시뮬레이션**: 모의 가격을 -7% / +15% / 고점 후 -5%로 주입해 각 규칙이 정확한 수량으로 체결되는지 확인 (특히 익절 후 잔량이 5,000원 미만인 경계 케이스).
5. **예산 소진 시나리오**: `api_usage`에 $8 / $10 상당을 인위 삽입 → 감속·정지 전환 확인, **정지 상태에서도 손절이 작동하는지** 확인 (가장 중요한 안전 검증).
6. **평가 전 쿨다운 검증**: 시스템 시간 조작(테스트에서 주입)으로 쿨다운 중 매수 주문이 거부되는지 확인.
7. **실거래 전환**: 5,000원 1회 수동 주문 → 체결·잔고·대시보드 반영 3중 확인 후 본 가동.

## 12. 구현 순서 (마일스톤)

1. **M1 — 골격**: config, store(스키마 5종), upbit_client(시세 + DRY_RUN 모의 체결)
2. **M2 — 안전장치 우선**: watchdog(청산 규칙 4종) + budget(예산 관리) — **판단보다 먼저 만든다**. 안전장치 없는 봇은 dry-run이어도 코드 신뢰가 안 쌓임
3. **M3 — 두뇌**: strategist(프롬프트 + tool use) → executor(가드레일) → 사이클 1회 수동 실행 성공
4. **M4 — 자동화**: scheduler(동적 주기 + 1분 감시 + 스냅샷 + 주간 평가), judge
5. **M5 — 대시보드**: FastAPI + 5개 화면 + PWA
6. **M6 — 검증**: 테스트 스위트 + dry-run 개시
7. **M7 — 실전**: 업비트 키 연동 → 최소 실거래 → VPS 이전
